# 深度分析视角

这个文件只在用户明确要求往深处拆的时候再读。

主用途不是普通改写，主用途是避免分析停在表层。

## 先看什么

用户要深度分析时，优先看这些对象：

1. 知识对象
2. 执行链路
3. 责任边界
4. 权限条件
5. 人工接管点
6. 审计记录
7. 成本单元
8. 失败路径

## 不要停在哪

很多分析会停在这些层面：

1. 工具层
2. 协议层
3. 模型层
4. 产品表面功能

这些层面可以提，但不要来回复述。

## 往下拆的方法

### 1. 知识对象

先问资料本身完不完整。

看这些点：

1. 文档写的是结论，还是写了判断过程。
2. 文档有没有例外条件。
3. 文档有没有真实操作口径。
4. 文档有没有归属人和更新机制。

### 2. 执行链路

再问任务怎么流转。

看这些点：

1. 输入从哪里来。
2. 中间经过哪些步骤。
3. 哪一步在调用工具。
4. 哪一步在等待人工确认。
5. 哪一步最容易断。

### 3. 责任边界

再问谁负责什么。

看这些点：

1. 哪些判断以前靠人。
2. 哪些裁量空间没有写进系统。
3. 出错以后谁来接。
4. 结果偏了以后谁来追。

### 4. 权限条件

再问执行动作受什么条件限制。

看这些点：

1. 系统权限是不是静态授权。
2. 业务动作是不是依赖情境判断。
3. 规则有没有写成机器能执行的条件。
4. 人工确认点放得够不够前。

### 5. 成本单元

再问成本是怎么长出来的。

看这些点：

1. 模型调用只是账单的一部分。
2. 长上下文会抬成本。
3. 工具等待和重试会抬成本。
4. 人工复核和失败补偿也会抬成本。

## 面向企业写建议时

建议不要停在“继续优化模型”这种层面。

更实用的写法通常是：

1. 先挑哪类任务。
2. 先补哪类状态记录。
3. 先立哪类权限边界。
4. 先放哪类人工接管点。
5. 先算哪类任务成本。

## 面向学习者写建议时

建议不要停在“多学框架、多学模型”这种层面。

更实用的写法通常是：

1. 先学任务建模。
2. 先学状态记录。
3. 先学工具调用。
4. 先学引用和评测。
5. 先做单 agent 项目。
