## 合格输出的特点

合格输出通常会做到这些事：

1. 先把 AI 味最重的地方找出来。
2. 删除套话、空词、金句化句子、宣传口吻。
3. 保留原意，不做额外拔高。
4. 原文重点完整保留。
5. 句子读起来像正常中文或正常英文，不像模型口播。
6. 用户要求分析时，能继续往系统对象和执行链路去写。
7. 用户要求动作时，能给企业动作或学习动作。
8. 时间敏感事实带具体日期和直接链接。
9. 引用点和原文链接能一一对应。
10. 通过禁用词检查。
11. 没有连续堆叠的三法则结构。
12. 没有无信号过渡词。
13. 英文改写以后，不像演讲稿，也不像品牌文案。

## 合格输出的形状

常见结构可以是这样：

1. 先点出原文问题。  
   例如：套话多、句子太工整、空词太多、结尾像口号。
2. 再列原文重点。
3. 再给改写结果。
4. 用户要求解释时，再补为什么这样改。
5. 用户要求分析时，再继续往知识对象、权限边界、任务状态、成本路径去写。
6. 用户要求建议时，最后补企业动作和学习动作。
7. 用户要求补来源时，把引用点和原文链接对应起来。

## 改写前后对比例子

### 例子 1：去 AI 味

改写前：

> 值得注意的是，AI 正在深刻改变企业软件的整体形态。它不仅提升了工作效率，还重塑了组织协作方式。对于企业而言，只有尽快完成认知升级，才能真正抓住这一轮技术红利。

改写后：

> AI 接进企业系统以后，变化先落在流程里。原来靠人补的判断、确认和解释，现在要写进系统。流程没理顺，模型再强也接不住。

说明：

1. 删掉“值得注意的是”这类空过渡词。
2. 删掉“深刻改变”“认知升级”“技术红利”这类空泛词。
3. 把话落回具体对象，直接写流程和判断。

### 例子 2：重点保留，顺序重排

原文重点：

1. 知识库质量问题不只在检索。
2. 企业文档常常缺判断过程和例外条件。
3. 人能补上下文，模型拿不到这些隐含信息。

改写前：

> 很多团队一直在调检索、调重排、调 embedding，但知识库效果还是不稳定。原因在于，企业文档写的往往只是结论，没有写判断过程，也没有写例外条件。人能看懂，是因为人会补上下文，模型不会。

改写后：

> 知识库效果不稳定，很多时候不是检索参数没调好。企业文档里经常只有结论，没有判断过程，也没有例外条件。人能把这类材料接着用下去，靠的是经验补全。模型拿不到这层信息，后面的检索和生成都会受影响。

说明：

1. 原文三条重点全部保留。
2. 顺序做了微调，先下判断，再展开原因。
3. 没额外拔高，也没补原文没有的立场。

### 例子 3：去三法则堆叠

改写前：

> 产品解决问题，平台创造空间，生态定义未来。系统管理流程，连接数据，驱动协同。团队需要速度、精度和规模。

改写后：

> 这段话的问题不在观点，在写法。三组整齐句连着摆，信息密度反而下去了。改成正常说法会更清楚：产品先解决眼前问题。平台把能力接到更多流程里。团队接下来要处理的，是速度、准确率和运维成本。

说明：

1. 拆掉连续堆叠的整齐句。
2. 留住原来的意思。
3. 补足原句里没有说清的对象。

### 例子 4：英文去 AI 味

改写前：

> Importantly, this approach not only improves operational efficiency, but also unlocks a more scalable collaboration model for modern teams. As organizations continue to evolve, this kind of capability will become increasingly essential.

改写后：

> This approach helps teams move work faster, and it makes handoffs easier to manage. The useful part is that fewer steps stay in people’s heads.

说明：

1. 删掉 `Importantly` 这种空提示词。
2. 删掉 `not only ... but also ...` 这种现成结构。
3. 删掉 `unlocks`、`increasingly essential` 这类宣传味写法。
4. 把英文落回具体动作和具体变化。

## 引用和链接例子

### 例子 5：引用点和原文链接对齐

不合格：

> OpenAI 最近在加强 agent 能力。
>
> 参考资料：
> - https://openai.com/index/new-tools-and-features-in-the-responses-api/

合格：

> OpenAI 在 2025 年 5 月 21 日写明 `Responses API` 增加了 “support for all remote Model Context Protocol (MCP) servers” 和 `background mode`。这说明远程工具接入和长任务执行已经进入主路径。  
> 链接：[OpenAI Responses API 更新](https://openai.com/index/new-tools-and-features-in-the-responses-api/)

说明：

1. 引用点和链接放在一起。
2. 写出原文关键信息，不只丢网址。
3. 时间敏感内容写具体日期。

## 不合格输出

避免写成下面这样：

1. 只把词换一换，AI 味还在。
2. 把原文改成另一种更重的套话。
3. 一开口就摆“资深架构师”人设。
4. 堆很多抽象词，却没有具体问题落点。
5. 出现自媒体、鸡汤、行情、文案化句式。
6. 为了顺口删掉原文里的关键判断。
7. 链接放在文末，正文里看不出对应关系。
8. 一段里连续摆好几组三法则结构。
9. 用“值得注意的是”“重要的是”“有趣的是”硬开新段。
10. 英文里全是 `Importantly`、`Notably`、`as we move forward` 这一类空提示词。
11. 英文里整段像演讲稿、路演稿、品牌文案。

## 请求类型和动作映射

### 请求类型：去 AI 味改写

动作重点放在这些地方：

1. 先找 AI 痕迹最重的句子。
2. 先删模板化结构和空词。
3. 保留原意和信息量。
4. 让句子恢复正常口语。
5. 必要时给改写前后对比。
6. 原文重点先列出来，再动笔。
7. 连续三段式先拆。
8. 无信号过渡词先删。

### 请求类型：重点保留但顺序可调

动作重点放在这些地方：

1. 先列原文重点。
2. 再判断哪些段落能重排。
3. 顺序调整以后检查有没有丢核心判断。
4. 小标题和分段一起整理。

### 请求类型：补文章引用

动作重点放在这些地方：

1. 先确认引用点。
2. 再补原文链接。
3. 引用内容和链接放在一起。
4. 时间敏感资料补具体日期。

### 请求类型：改写前后对比

动作重点放在这些地方：

1. 先列原文问题。
2. 再给改写结果。
3. 说明删了什么、保了什么、为什么这样改。
4. 不要只给结论，不给改写依据。

### 请求类型：去三法则和空过渡词

动作重点放在这些地方：

1. 先找连续成组的三段式。
2. 再找“值得注意的是”“重要的是”“有趣的是”这类空过渡词。
3. 把整齐句拆开。
4. 把提示词删掉后重接句子。

### 请求类型：企业应该怎么做

动作重点放在这些地方：

1. 起步任务怎么选。
2. 状态和日志怎么立。
3. 权限和人工接管怎么放。
4. 哪些写操作先不要进生产。
5. 任务成本怎么算。

### 请求类型：全栈转 agent 开发怎么学

动作重点放在这些地方：

1. 先学任务建模。
2. 再学工具调用和状态管理。
3. 项目里一定要有日志、评测、引用、人工确认。
4. 先做单 agent 项目。
5. 再碰多 agent 或跨系统执行。

### 请求类型：英文去 AI 味改写

动作重点放在这些地方：

1. 先删英文里的空提示词。
2. 先拆 `not only ... but also ...`、`it is important to note that ...` 这类现成结构。
3. 删掉品牌文案味和演讲稿味。
4. 保住原句判断，不把英文改成另一种更假、更满的英文。
